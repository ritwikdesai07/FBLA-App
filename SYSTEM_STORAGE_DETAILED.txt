FBLA APP - DETAILED SYSTEM + STORAGE DOCUMENT

Purpose of this document:
This explains exactly how the FBLA app works internally, with specific focus on system structure, routing control, persistence model, and data flow between screens.

==================================================
1) CORE APP STACK AND RUNTIME MODEL
==================================================

Framework/runtime:
- React Native app built with Expo (package: expo ^54.0.31)
- Routing uses expo-router (~6.0.22), file-based route mapping
- TypeScript project
- Local persistence uses AsyncStorage (@react-native-async-storage/async-storage 2.2.0)

Important architectural point:
- This is a local-first app (no remote backend/database/API).
- All auth/user/reminder/profile data exists on-device in AsyncStorage.

Main app shell files:
- app/_layout.tsx
- app/(tabs)/_layout.tsx
- lib/authStorage.ts

==================================================
2) ROUTING SYSTEM (EXPO ROUTER) AND ACCESS CONTROL
==================================================

2.1 Root layout behavior (app/_layout.tsx)
- Wraps app in ThemeProvider from app/(tabs)/ThemeContext.tsx.
- Wraps navigation with @react-navigation/native ThemeProvider (DarkTheme/DefaultTheme based on device scheme via hooks/use-color-scheme.ts).
- Creates stack with:
  - (tabs) route group (header hidden)
  - modal route
- Sets unstable_settings.anchor = '(tabs)' so tabs act as navigation anchor.

2.2 Tab layout gatekeeper (app/(tabs)/_layout.tsx)
This file is the main auth/profile gate. It computes authFlowState:
- 'loading'
- 'unauthenticated'
- 'needs-setup'
- 'ready'

State logic:
- Calls getCurrentUser() from lib/authStorage.ts.
- If no user => unauthenticated.
- If user exists but user.profile?.profileComplete is false/undefined => needs-setup.
- If profileComplete true => ready.

Route enforcement:
- unauthenticated users are redirected to /landing unless already on auth pages.
- needs-setup users are forced to /profile-setup.
- ready users are redirected away from /profile-setup back to /.

Tab sets by state:
- Unauthenticated:
  landing, auth-choice, login, signup (tab bar hidden)
- Needs-setup:
  profile-setup only (tab bar hidden)
- Ready:
  index (Home), reminders, dashboard, other
  plus hidden routes for auth/setup pages (href: null)

Realtime refresh behavior:
- _layout.tsx uses setInterval(load, 1000)
- This means auth/profile state is reloaded every 1 second.

==================================================
3) AUTH + DATABASE LAYER (lib/authStorage.ts)
==================================================

This file is the single source of truth for account and data persistence.

3.1 Storage keys
- USERS_DB_KEY = '@fbla_users_v1'
- CURRENT_USER_KEY = '@fbla_current_user_v1'

3.2 Database structures (exact conceptual schema)

Reminder type:
- Reminder = { id: string; title: string; notes: string }

Reminder map by date:
- ReminderMap = Record<string, Reminder[]>
  where key is date string in YYYY-MM-DD format

Calendar scopes:
- CalendarType = 'FBLA National' | 'FBLA State' | 'FBLA Regional'

All reminders grouped by scope:
- AllReminders = Record<CalendarType, ReminderMap>

User profile shape:
- profile?: {
    state?: string;
    school?: string;
    chapterName?: string;
    gradDate?: string;        // YYYY-MM-DD
    profileImageUri?: string | null;
    duesTotal?: number;
    duesPaid?: number;
    profileComplete?: boolean;
  }

Full user record:
- UserRecord = {
    username: string;
    password: string;
    displayName: string;
    reminders: AllReminders;
    profile?: ...
  }

Database container:
- UsersDb = Record<string, UserRecord>
- Key is normalized username (trim + lowercase)

3.3 Username normalization
- normalizeUsername(username) => username.trim().toLowerCase()
- All signups/logins use normalized username, so case variants map to same account.

3.4 CRUD/auth functions and exact effects

loadUsersDb():
- Reads @fbla_users_v1 from AsyncStorage.
- If missing => {}.
- If JSON parse fails => {} (silent recovery behavior).

saveUsersDb(db):
- Serializes and writes full db back to @fbla_users_v1.

emptyReminders():
- Returns object with all three calendar scopes pre-created:
  {
    'FBLA National': {},
    'FBLA State': {},
    'FBLA Regional': {}
  }

signUpUser(username, password, displayName):
- Validates non-empty username/password.
- Rejects duplicate normalized username.
- Creates UserRecord with:
  - username normalized
  - password (trimmed)
  - displayName fallback to username
  - reminders initialized via emptyReminders()
  - default profile with profileComplete: false
- Persists db.
- Sets CURRENT_USER_KEY to new username (auto-login after signup).

loginUser(username, password):
- Normalizes username and trims password.
- Loads user from db.
- Checks exact password match.
- If valid: sets CURRENT_USER_KEY.

logoutUser():
- Removes CURRENT_USER_KEY only.
- Does not delete user database.

getCurrentUser():
- Reads CURRENT_USER_KEY.
- Loads db.
- Returns db[currentUsername] or null.

setCurrentUserReminders(reminders):
- Requires logged-in user.
- Replaces current user's entire reminders object with provided one.
- Saves updated db.

setCurrentUserProfile(profilePatch):
- Requires logged-in user.
- Merges patch into existing profile:
  profile = { ...existingProfile, ...profilePatch }
- Saves updated db.

setCurrentUserDisplayName(displayName):
- Requires logged-in user.
- Trims and updates displayName (fallback to existing if blank).
- Saves updated db.

Security note (important for presentation honesty):
- Passwords are stored in plaintext in AsyncStorage.
- Good for prototype/local demo, not production security.

==================================================
4) AUTHENTICATION + PROFILE FLOW (SCREEN LEVEL)
==================================================

4.1 landing.tsx
- Marketing entry screen.
- "Get Started" => router.replace('/auth-choice')

4.2 auth-choice.tsx
- Branch point:
  - Login => /login
  - Sign Up => /signup

4.3 login.tsx
- Collects email (used as username) + password.
- Calls loginUser(email.trim(), password).
- On success => router.replace('/')
- Tab layout then decides final destination by authFlowState.

4.4 signup.tsx
Collects:
- email
- password
- displayName
- state (Picker from full US state list)
- school
- chapterName
- gradDate (Calendar modal)

Save sequence:
1) signUpUser(...) creates account with profileComplete false
2) setCurrentUserProfile({... profile fields ..., profileComplete: true, duesTotal: 0, duesPaid: 0})
3) router.replace('/')

4.5 profile-setup.tsx (fallback gate)
- Used when logged in but profileComplete is false.
- Preloads current user fields.
- Requires all profile fields.
- On save:
  - setCurrentUserDisplayName(...)
  - setCurrentUserProfile(..., profileComplete: true)
- Then redirects to '/'.

==================================================
5) REMINDER SYSTEM (DATA MODEL + UI LOGIC)
==================================================

Primary file:
- app/(tabs)/reminders.tsx

5.1 State model in reminders screen
- selectedType: CalendarType
- allReminders: AllReminders
- selectedDate: string (YYYY-MM-DD)
- formData: { title, notes }

5.2 Calendar interaction
- onDayPress(day):
  - set selectedDate
  - open modal in list mode
- onDayLongPress(day):
  - set selectedDate
  - open modal in add mode

5.3 Marked dates logic
- Builds markedDates from keys of currentReminders[selectedType].
- Adds dot markers for dates with reminders.
- Highlights selected date.

5.4 Reminder creation
- Requires selectedDate and non-empty title.
- New reminder object:
  {
    id: Date.now().toString(),
    title: trimmed title,
    notes: trimmed notes
  }
- Writes into nested structure:
  allReminders[selectedType][selectedDate] = existing + newReminder
- Persists with setCurrentUserReminders(updatedState)

5.5 Reminder deletion
- Removes by reminder id from day list.
- If day becomes empty, deletes date key from that calendar map.
- Persists with setCurrentUserReminders(updatedState)

==================================================
6) HOME TAB SUMMARY SYSTEM (app/(tabs)/index.tsx)
==================================================

6.1 Data loading
- Reads getCurrentUser() every 1 second.
- Loads user + all reminders.

6.2 Upcoming reminders algorithm
- Defines today at 00:00 local time.
- Defines oneWeekAway = today + 7 days.
- Iterates all 3 calendar scopes.
- For each date key (YYYY-MM-DD), parses local date.
- Includes reminders where date in [today, oneWeekAway].
- Flattens and sorts by date ascending.

6.3 Dashboard metrics shown on home
- Upcoming count = number of reminders in next 7 days.
- Dues percent:
  if duesTotal > 0 => round((duesPaid / duesTotal) * 100), clamped 0..100
  else 0.

==================================================
7) DASHBOARD TAB SYSTEM (app/(tabs)/dashboard.tsx)
==================================================

7.1 Profile display
- Shows read-only fields from user.profile:
  state, gradDate, school, chapterName
- Display name from user.displayName

7.2 Dues update modal logic
Inputs:
- duesToAdd (numeric text)
- paidToAdd (numeric text)

Computation:
- totalAdd = max(0, Number(duesToAdd) || 0)
- paidAdd = max(0, Number(paidToAdd) || 0)
- updatedTotal = existingTotal + totalAdd
- updatedPaid = min(updatedTotal, existingPaid + paidAdd)

Writes:
- setCurrentUserProfile({ duesTotal: updatedTotal, duesPaid: updatedPaid })

7.3 Logout
- Calls logoutUser()
- router.replace('/auth-choice')

==================================================
9) POLLING / REFRESH STRATEGY
==================================================

Files with periodic reload behavior use setInterval(load, 1000):
- app/(tabs)/_layout.tsx
- app/(tabs)/index.tsx
- app/(tabs)/reminders.tsx
- app/(tabs)/dashboard.tsx
- app/(tabs)/home.tsx (alternate/home variant)

Why this matters:
- UI updates quickly after data writes without centralized state manager.
- Tradeoff: repeated reads every second can be less efficient than event-driven state.

==================================================
11) SYSTEM LIMITATIONS (TECHNICAL)
==================================================

- No backend sync: data does not automatically move across devices.
- Plaintext password storage: not secure for production.
- Polling-based refresh: simple but not optimal for performance/scalability.
- AsyncStorage data could be cleared if app/device storage is reset.

12) HIGH-QUALITY PRESENTATION SUMMARY (SHORT SCRIPT)
==================================================

"Our FBLA app is a local-first Expo React Native system using file-based routing and a custom auth gate in the tabs layout. The app stores a full user database in AsyncStorage under @fbla_users_v1 and tracks the active session with @fbla_current_user_v1. Each user record contains credentials, profile metadata, dues, and reminders indexed first by FBLA level and then by date. Reminders are created and deleted through a calendar UI and persisted by replacing the user reminder map. Access control is state-driven: unauthenticated users are routed to landing/auth, incomplete profiles are forced through profile setup, and only completed profiles reach the main tabs. Home computes weekly reminders and dues progress from persisted data, while dashboard manages profile snapshot and dues accumulation. This architecture is strong for offline demos and rapid iteration, with clear next steps to move auth and persistence to a secure cloud backend."
