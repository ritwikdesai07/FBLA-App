FBLA App — Systems & Storage (Detailed)

Overview
- This document explains how the app organizes data, how user accounts and authentication work, and exactly where and how information is saved on device.

Key storage primitives
- The app uses React Native's `AsyncStorage` (imported from `@react-native-async-storage/async-storage`) as the local persistent storage mechanism.
- Two top-level keys are used in `lib/authStorage.ts`:
  - `@fbla_users_v1` — the serialized Users database (JSON). Type `UsersDb = Record<string, UserRecord>`.
  - `@fbla_current_user_v1` — the username (string) of the currently logged-in user.

Data shapes (from `lib/authStorage.ts`)
- `UserRecord` fields:
  - `username: string`
  - `password: string` (stored in plaintext in the DB; see security notes)
  - `displayName: string`
  - `reminders: AllReminders` — calendar-based reminders (nested objects)
  - `profile?: { state?, school?, chapterName?, gradDate?, profileImageUri?, duesTotal?, duesPaid?, profileComplete? }

- `AllReminders` is a nested map keyed by calendar type (e.g., 'FBLA National') then by date IDs; empty by default via `emptyReminders()`.

How sign up works (`signUpUser`)
1. Input: `username`, `password`, `displayName`.
2. Normalization: `username` is trimmed and lowercased; `password` is trimmed.
3. Validation: requires non-empty username and password.
4. Load DB: reads `@fbla_users_v1` and parses JSON. If missing or invalid, an empty object is used.
5. Duplicate check: returns error if username already exists.
6. Save new user record in the DB object with default `reminders` and an initial `profile` object.
7. Persist DB: writes the full DB back to `@fbla_users_v1` using `AsyncStorage.setItem` with `JSON.stringify(db)`.
8. Persist current user: stores clean username in `@fbla_current_user_v1`.
9. Returns `{ ok: true }` on success or `{ ok: false, error }` on failure.

How login works (`loginUser`)
1. Input: `username`, `password` from the login UI (see `app/(tabs)/login.tsx`).
2. Normalization: username is trimmed + lowercased; password is trimmed.
3. Load DB: reads `@fbla_users_v1` and parses JSON.
4. Lookup: find `db[cleanUsername]`.
5. Password check: compares stored `user.password` exactly to the provided password string.
   - If no user or mismatch, returns `{ ok: false, error: 'Invalid username or password.' }`.
6. On success, writes `@fbla_current_user_v1 = cleanUsername` so `getCurrentUser()` can restore the session.
7. The login screen calls `loginUser(...)`; on success it calls `router.replace('/')` to navigate to the app root.

How logout works (`logoutUser`)
- Removes `@fbla_current_user_v1` from `AsyncStorage` using `removeItem`.
- The app checks `getCurrentUser()` on boot or in guarded screens to redirect to auth flows when null.

How current user is retrieved (`getCurrentUser`)
1. Read `@fbla_current_user_v1`. If empty, return `null` (no session).
2. Read `@fbla_users_v1`, parse JSON, return `db[currentUsername] ?? null`.

How profile & reminders are saved
- `setCurrentUserProfile(profilePatch)`:
  - Reads current username from `@fbla_current_user_v1`.
  - Loads the full DB, merges `profilePatch` into `user.profile` and overwrites the user record.
  - Persists the entire DB back to `@fbla_users_v1`.

- `setCurrentUserReminders(reminders)`:
  - Reads current username, loads DB, replaces `user.reminders` with the provided object and persists DB.

- `setCurrentUserDisplayName(displayName)` updates the `displayName` field and persists.

Where the UI hooks into auth storage
- `app/(tabs)/signup.tsx` calls `signUpUser(...)`, and on success calls `setCurrentUserProfile(...)` then navigates to `/`.
- `app/(tabs)/login.tsx` calls `loginUser(...)`, and on success `router.replace('/')`.
- `app/(tabs)/profile-setup.tsx` uses `getCurrentUser()` on `useEffect` to redirect or prefill fields; `handleSave()` calls `setCurrentUserDisplayName` and `setCurrentUserProfile`.

Security notes and risks (important)
- Passwords are stored in plaintext inside `@fbla_users_v1`. This is insecure for production.
- `AsyncStorage` is not encrypted; its contents can be read on a rooted/jailbroken device or from backups.
- The app appears to be an offline-only local-auth model (no server). If that's intentional, consider the following mitigations:
  - Never store raw passwords. Instead store a salted hash (e.g., using bcrypt) or derive a key via PBKDF2 and store the derived value.
  - For sensitive data or tokens, use a secure store: `expo-secure-store`, `react-native-keychain`, or platform keystores.
  - If you plan to support multi-device sync, migrate to a server-backed auth flow with tokens (e.g., JWT) and avoid storing credentials locally.

Data model & migration guidance
- Users DB versioning: the keys use a `_v1` suffix (`@fbla_users_v1`) — when changing the DB shape, increase the suffix (e.g., `_v2`) and add a migration script to read `_v1`, transform records, and write `_v2`.
- Because the app writes the entire DB object on each change, concurrent edits or partial writes could cause data loss. To harden:
  - Consider writing per-user keys (`@fbla_user_<username>`) rather than a single monolithic `@fbla_users_v1` JSON blob.
  - Make writes atomic and consider a backup copy before overwriting (write `tmp` then rename pattern).

Practical tips for maintainers
- To change storage keys or behavior, edit `lib/authStorage.ts`:
  - `USERS_DB_KEY` and `CURRENT_USER_KEY` are the single source-of-truth keys.
  - All read/write helpers are centralized there.
- To change UI flows that trigger auth actions, edit `app/(tabs)/login.tsx`, `signup.tsx`, and `profile-setup.tsx`.
- To inspect stored data on a device/emulator, use platform tooling to view AsyncStorage (or add a small debug screen that reads and prints `@fbla_users_v1`).

Recommended short-term improvements
- Replace plaintext password storage with a secure hash (bcrypt or Argon2 via a native module, or PBKDF2).
- Replace storing entire DB in one key with per-user keys, or use a small local database (SQLite) for structured updates.
- Move very sensitive fields to SecureStore.
